# Руководство по установке и использованию Edge B.R.O.

🇬🇧 **[English README](README.md)** | 🇬🇧 **[English Usage Guide](README_USAGE.md)** | 🇷🇺 **[Русский README](README_ru.md)** | 🇷🇺 **[Русская инструкция (Usage Guide)](README_USAGE_ru.md)**

Данный документ описывает полный цикл развертывания системы оркестрации резервного копирования и восстановления, начиная с "голого" ПК (например, Intel NUC), и заканчивая процессом восстановления (Bare-Metal Restore) на физические диски.

---

## 1. Подготовка сервера (Intel NUC)

В качестве центрального сервера управления (оркестратора) отлично подойдет компактный ПК, такой как Intel NUC.

1. Установите на него базовую ОС Linux (например, Ubuntu 22.04/24.04 или Debian 12).
2. Установите **Docker** и **Docker Compose**:
   ```bash
   sudo apt update
   sudo apt install -y docker.io docker-compose-v2
   sudo systemctl enable --now docker
   sudo usermod -aG docker $USER
   ```
   *(После добавления пользователя в группу docker перезайдите в сессию).*

---

## 2. Развертывание системы оркестрации

1. Склонируйте репозиторий проекта:
   ```bash
    git clone https://github.com/masseselsev/Edge.bro.git /opt/stacks/Edge.bro
    cd /opt/stacks/Edge.bro
   ```

2. Создайте файл `.env` в корневой папке проекта, скопировав шаблонный файл и отредактировав его. Обязательно задайте надежные пароли:
   ```bash
   cp .env.example .env
   ```
   **Важный параметр `ORCHESTRATOR_IP`**: если ваши комплексы (клиентские узлы) находятся в другой подсети или подключаются через VPN/ZeroTier, укажите здесь IP-адрес самого NUC, по которому он доступен для клиентских машин. По этому IP узлы будут отправлять данные в центральный Borg-репозиторий.
   ```env
   POSTGRES_USER=postgres
   POSTGRES_PASSWORD=your_secure_db_password
   POSTGRES_DB=borg_orchestrator
   REDIS_URL=redis://redis:6379/0
   BORG_PASSPHRASE=your_secure_borg_passphrase
   DATABASE_URL=postgresql://postgres:your_secure_db_password@db:5432/borg_orchestrator
   ORCHESTRATOR_IP=192.168.222.2  # IP-адрес оркестратора, доступный для комплексов
   
   # Параметры стабильности SSH-соединения Keepalive (опционально, указаны дефолтные значения)
   # SSH_KEEPALIVE_INTERVAL=30    # Интервал отправки пакетов проверки связи (в секундах)
   # SSH_KEEPALIVE_COUNT=3       # Лимит неотвеченных запросов до разрыва соединения
   ```
   *(Примечание: в дальнейшем вы сможете переопределить этот адрес "на лету" прямо через веб-интерфейс во вкладке **Settings**, без необходимости править файл .env и перезапускать контейнеры).*

3. **Настройка места хранения бэкапов (Важно!)**
   По умолчанию все репозитории сохраняются во внутренний volume Docker (`borg-data`), который физически находится в `/var/lib/docker/volumes/`. Так как бэкапы могут занимать много места, рекомендуется вынести их на отдельный емкий диск, чтобы не переполнять системный корневой раздел.

   > [!WARNING]
   > Нехватка свободного места на корневом разделе приведет к сбоям задач резервного копирования с ошибками `Insufficient free space to complete transaction` и может нарушить работу других служб хоста.
   > 
   > *Примечание: Оркестратор автоматически выполняет `borg compact` после удаления или очистки архивов для немедленного высвобождения дискового пространства, но использование выделенного раздела диска все еще критически важно.*

   Чтобы примонтировать внешнюю папку (например, `/mnt/hdd/borg_data`), настройте переменную `BORG_HOST_DATA_PATH` в файле `.env`:
   ```env
   BORG_HOST_DATA_PATH=/mnt/hdd/borg_data
   ```

   > [!IMPORTANT]
   > При использовании внешнего пути на хосте убедитесь, что папка существует на хост-машине и имеет правильные права доступа (`chown -R 1000:1000`), чтобы контейнеры Docker могли производить в неё запись:
   > ```bash
   > mkdir -p /mnt/hdd/borg_data
   > chown -R 1000:1000 /mnt/hdd/borg_data
   > ```
   > Если значение переменной `BORG_HOST_DATA_PATH` оставлено по умолчанию как `borg-data`, Docker будет использовать стандартный именованный volume.

   Настроенный путь также будет динамически отображаться во вкладке **Settings** в интерфейсе управления.

4. Запустите контейнеры:
   ```bash
   docker compose up -d --build
   ```
   *(Примечание: Миграции базы данных выполняются автоматически на старте контейнера `backend`. Он подождет готовности PostgreSQL, применит миграции Alembic и запустит FastAPI).*

5. **Готово!** Интерфейс управления доступен в браузере по адресу `http://<IP-вашего-NUC>:7777`.

## 3. Подготовка узлов (Provisioning)

Перейдите во вкладку **Fleet** в веб-интерфейсе для добавления комплексов (edge nodes).

1. **Добавление (Add Node)**:
   - Вы можете вводить IP-адреса по одному, списком через запятую, диапазоном (например, `192.168.1.50-60`) или CIDR подсетью (`10.0.0.0/24`).
   - Укажите логин (обычно `root` или `admin`), пароль и SSH порт (например, `22` или `2222`).
2. **Провижининг (Bootstrap)**:
   - При добавлении оркестратор автоматически подключится к узлу, установит необходимые пакеты (даже если на узле прописан "мертвый" APT-прокси, система временно его обойдет) и пропишет свой публичный SSH-ключ (`/root/.ssh/authorized_keys`).
   - Дальнейшее взаимодействие с узлом будет происходить **без пароля**, по безопасному SSH-ключу оркестратора.
3. **Настройка дисков (Preparation)**:
   - В процессе работы система читает текущую структуру диска комплекса, сохраняет уникальный `EFI UUID` и развешивает постоянные метки (labels) файловой системы (`edgeroot`, `edgeboot`). Это страхует систему от смещения имен дисков в Linux.

## 4. Создание бэкапа

1. В строке нужного комплекса во вкладке **Fleet** нажмите кнопку **Backup**.
2. В появившемся окне можно нажать "View Logs" для просмотра потока выполнения в реальном времени.
3. Оркестратор подключится к узлу по SSH и запустит команду бэкапа от лица клиента (push-модель), направляя поток данных в центральный контейнер `borg-server`.
4. Системные директории и настроенные пути исключаются динамически на основе глобальных настроек (задаются во вкладке настроек **Settings** в веб-интерфейсе).
   - **Список исключений по умолчанию**: `/dev/*,/proc/*,/sys/*,/run/*,/mnt/*,/media/*,/lost+found,/var/log/edge/*,/var/opt/edge/*`
5. За прогрессом и итоговым размером можно следить во вкладке **History**.

### 4.1 Лимиты ресурсов и оптимизация бэкапов

Для снижения сетевой нагрузки на каналы связи и предотвращения утилизации процессора на удаленных узлах, вы можете настраивать индивидуальные ограничения ресурсов при создании или редактировании **группы бэкапа (Backup Group)**:
* **Лимит скорости (Upload Rate Limit)**: Максимальная пропускная способность сети (в KiB/s) для потока резервной копии.
* **Ограничение CPU (CPU Quota)**: Предельная доля процессорного времени (от 0 до 400% от одного ядра CPU), выделяемая под процессы резервного копирования. Ограничение применяется непосредственно на стороне клиента при помощи `systemd-run --scope -p CPUQuota=...`.
* **Сжатие (Compression)**: Выбор конкретного алгоритма сжатия Borg (например, `lz4`, от `zstd:1` до `zstd:9`) для группы.
* **Интервал создания чекпоинтов (Checkpoint Interval)**:
  - По умолчанию система **автоматически рассчитывает** оптимальный интервал сброса кэша на диск на основе лимита скорости (например, чекпоинт каждые ~50 МБ данных при лимите скорости <= 500 KiB/s или каждые ~200 МБ при <= 5000 KiB/s), чтобы минимизировать потери прогресса бэкапа на медленных линиях при случайных обрывах связи.
  - При необходимости вы можете переопределить это поведение и задать интервал (в секундах) вручную.
* **Переопределение политики удержания (Retention Policy Override)**: Позволяет настроить индивидуальные правила удаления старых бэкапов для конкретной группы вместо глобальной политики. Система поддерживает три режима ротации:
  - **Интервальный (Interval)**: Сохранение заданного числа ежедневных, еженедельных и ежемесячных копий.
  - **По количеству (Count)**: Сохранение строго фиксированного числа последних архивов (например, хранить только последние 5 бэкапов).
  - **По временному окну (Timeframe)**: Сохранение всех архивов за указанный период (например, за последние 3 месяца).

Глобальные настройки для ограничений CPU по умолчанию, метода сжатия по умолчанию и глобальной политики ротации настраиваются во вкладке **Settings**.

### 💡 Как работают глобальная дедупликация и сжатие (Экономия места)
Поскольку все комплексы резервируются в **один общий центральный репозиторий** (`/data/borg/fleet`), встроенные механизмы дедупликации и сжатия Borg работают "насквозь" по всем устройствам сразу. 
Это значит, что одинаковые системные файлы (ядро Linux, пакеты Debian, библиотеки, докер-образы приложения), присутствующие на десятках разных устройств, будут физически сохранены на диске сервера **только один раз**, а все сохраненные чанки дополнительно сжимаются.

**Примерная экономия (при базовом размере ОС комплекса ~6 ГБ):**
* **1-й комплекс (Сжатие и внутренняя дедупликация)**: с ходу экономит **55% - 65%** дискового пространства благодаря архивированию Borg (по умолчанию `lz4`), сжимая 6 ГБ системы до ~2.2 - 2.7 ГБ фактического объема на диске.
* **Каждая последующая копия схожей системы (Сквозная дедупликация)**: экономит до **97%** для клонов и идентичных устройств (добавляя всего **~100 - 200 МБ** для того же устройства под другим хостнеймом) и около **20% - 30%** для комплексов с небольшими различиями в пакетах и конфигурации.
* **Итоговые инкрементные бэкапы**: работающих систем стремятся к небольшому объему около **~100-200 МБ** уникальных инкрементных данных на один запуск бэкапа (сохраняются только изменившиеся логи, кэши, базы данных).

В результате, для парка однотипных устройств на Debian достигается радикальное сокращение требований к дисковому пространству, а последующие инкрементные бэкапы остаются крайне легкими (~100-200 МБ).

## 5. Процедура восстановления (Bare-Metal Restore)

Эта функция позволяет раскатать резервную копию прямо на физический диск, подключенный к оркестратору (NUC) по USB.

### 5.1 Подключение диска к оркестратору
Для удобства можно использовать внешний переходник (донгл) **USB to SATA** или **USB to NVMe**.

⚠️ **Правила подключения накопителей**:
* **SATA диски**: Переходники USB-SATA обычно поддерживают горячее подключение (hot-plug). Вы можете подключить SSD диск к переходнику, а затем вставить USB в NUC прямо "на горячую".
* **NVMe диски**: Большинство переходников M.2 NVMe to USB **не поддерживают** горячее извлечение самого накопителя из колодки под напряжением. Всегда сначала вставляйте NVMe накопитель в переходник, и только затем подключайте кабель USB в NUC. При извлечении обязательно дождитесь завершения записи, выполните безопасное извлечение и только потом выдергивайте USB-кабель. Строго не рекомендуется вынимать M.2 плату из переходника, пока он вставлен в USB.

### 5.2 Процесс прошивки (Flashing)
1. В интерфейсе перейдите во вкладку **Flasher**.
2. **Target Device**: В правой части экрана выберите подключенный USB-диск. Система отображает физические метки дисков (модель, размер) и тип шины (SATA/NVMe).
   *(Системный диск самого NUC отфильтрован и защищен от случайной перезаписи, но будьте внимательны).* 
3. **Select Node & Archive**: В левой части выберите узел (комплекс), который вы хотите восстановить, и нужный снимок (архив) из списка.
   *(💡 **Миграция между типами дисков (NVMe ↔ SATA)**: За счет того, что система привязывает диски строго по меткам (`LABEL=edgeroot`) вместо аппаратных путей, вы можете свободно восстановить бэкап, снятый с NVMe-накопителя, на обычный SATA-диск, или наоборот. При несовпадении типов дисков интерфейс выдаст предупреждение "Drive type mismatch" (by-design) для предотвращения случайной ошибки оператора. Вы можете смело проигнорировать его и подтвердить операцию).*
4. Нажмите **Start Flashing**. 
   Откроется консоль логов. Вы увидите весь процесс:
   - Диск будет полностью очищен (wipefs) и переразмечен в GPT-таблицу.
   - Будут созданы разделы: EFI (`vfat`) и системные (`ext4`). Для `ext4` отключается несовместимая фича `orphan_file`, что гарантирует загрузку системы даже на старом Debian 10.
   - Восстановится исторический EFI UUID загрузочного раздела (чтобы материнская плата узла распознала загрузчик).
   - Файлы из архива Borg распакуются на смонтированный диск.
   - Выполнится привязка системных каталогов (`chroot`) и обновится загрузчик GRUB.
   - Сетевые настройки будут сброшены (удалены правила persistent-net-rules), и будет внедрена общая DHCP-конфигурация (для `eth*` и `en*`). Это гарантирует, что при установке диска в новое железо (где изменятся MAC-адреса и имена сетевых карт), комплекс автоматически получит IP-адрес.
5. После появления сообщения `Restore completed successfully!`, диск безопасно отмонтирован.
6. Отключите USB-переходник от NUC, извлеките диск и установите его в целевой комплекс. Включите комплекс — он загрузится полностью рабочим, со всеми данными на момент снятия бэкапа.

## 6. Восстановление без извлечения дисков (Live-USB Kiosk)

Хотя в Разделе 5 описана процедура восстановления при прямом физическом подключении диска к NUC, вы также можете восстанавливать узлы **без извлечения их накопителей**, используя сгенерированный Live-USB.

### 6.1 Генерация Live-USB
1. Перейдите во вкладку **Live-USB Client** в веб-интерфейсе.
2. Введите IP-адрес оркестратора (NUC) и токен авторизации (API Authentication Token). Эти данные будут безопасно "запечены" внутрь ISO-образа, чтобы клиентский узел мог автоматически авторизоваться на сервере.
3. Нажмите **GENERATE LIVE-USB**. Оркестратор динамически вычислит точный размер файла, скачает базовый образ Debian testing, внедрит конфигурацию и скомпилирует специальный загрузочный `technician_client_v1.iso`.
4. Дождитесь завершения процесса, нажмите **DOWNLOAD ISO IMAGE** и прошейте скачанный образ на USB-флешку с помощью [Rufus](https://rufus.ie/ru/) или [balenaEtcher](https://etcher.balena.io/).

### 6.2 Использование Live-USB
1. Вставьте подготовленную USB-флешку в неисправный узел и загрузитесь с неё.
2. Убедитесь, что узел подключен к той же локальной сети, что и оркестратор.
3. Система автоматически загрузит легковесную среду Debian XFCE и запустит защищенный браузер-киоск.
4. Киоск сам подключится к центральному оркестратору и отобразит интерфейс восстановления (Flasher).
5. Выберите локальный диск самого узла и нужный снимок резервной копии. Процесс восстановления по сети скачает данные напрямую из репозитория и безопасно "прошьет" внутренний диск узла.

## 7. Детальный список изменений, вносимых оркестратором на клиентских узлах

Во время жизненного цикла оркестратор выполняет строго определенный набор неразрушающих операций на удаленных комплексах, устанавливая только необходимые для своей работы зависимости:

### На этапе подключения (Bootstrap)
1. **Обход неработающих прокси**: Система проверяет доступность прокси-серверов, указанных в `/etc/apt/apt.conf` и `/etc/apt/apt.conf.d/`. Если прокси доступен — он используется штатно. И только если прокси "мертв" (сервер выключен или недосягаем), происходит его временное отключение с последующим возвратом конфигурации по завершении установки пакетов.
2. **Установка системных пакетов**: Устанавливаются `python3` и `aptitude` (необходимы для базовой работы Ansible).
3. **Настройка SSH для беспарольного доступа**:
   - Создается директория `/root/.ssh` (с правами `0700`).
   - Публичный Ed25519-ключ оркестратора добавляется в `/root/.ssh/authorized_keys` (права `0600`).
   - В `/etc/ssh/sshd_config` параметр `PermitRootLogin` меняется на `prohibit-password` (разрешен вход root только по ключам, вход по паролю отключается).
   - Выполняется перезапуск сервиса `ssh` / `sshd`.

### На этапе подготовки дисков (Provisioning / Prepare)
1. **Установка утилит для работы с дисками и бэкапом**: Устанавливаются пакеты `e2fsprogs`, `parted`, `dosfstools`, `borgbackup`.
2. **Маркировка файловых систем** (с помощью утилиты `e2label`):
   - Корневому разделу (`/`) присваивается постоянная метка `edgeroot`.
   - Загрузочному разделу (`/boot`) присваивается метка `edgeboot`.
   - Разделу логов (`/var/log/edge`) присваивается метка `edgelog`.
   - Разделу данных (`/var/opt/edge`) присваивается метка `edgestor`.
3. **Обновление /etc/fstab**: Старые привязки дисков заменяются на строгий шаблон монтирования по созданным меткам (`LABEL=edgeroot`, `LABEL=edgeboot` и т.д.). Исключением является только раздел EFI (`/boot/efi`), который монтируется по своему "родному" аппаратному `UUID`.
4. **Сбор конфигурации**: Считывается и передается в БД оркестратора уникальный идентификатор EFI UUID комплекса.

### На этапе резервного копирования (Backup)
Новые пакеты на узле **не устанавливаются**. Оркестратор лишь запускает уже установленный клиент `borg create`, который:
1. Выполняет чтение файловой системы узла.
2. Автоматически **исключает** виртуальные, временные и настроенные директории комплексов на основе глобальных настроек оркестратора (список исключений по умолчанию: `/dev/*`, `/proc/*`, `/sys/*`, `/run/*`, `/mnt/*`, `/media/*`, `/lost+found`, `/var/log/edge/*`, `/var/opt/edge/*`).
3. Хеширует данные и передает только новые, дедуплицированные блоки по SSH на центральный `borg-server` оркестратора.
