# Edge B.R.O. — Edge Backup & Restore Orchestrator

🇬🇧 **[English README](README.md)** | 🇬🇧 **[English Usage Guide](README_USAGE.md)** | 🇷🇺 **[Русский README](README_ru.md)** | 🇷🇺 **[Русская инструкция (Usage Guide)](README_USAGE_ru.md)**

Контейнеризованная система оркестрации промышленного уровня для централизованного управления, планирования резервных копий и полного bare-metal восстановления для парка из множества edge-комплексов на базе Debian.

---

## 🏗️ Обзор архитектуры

Система полностью работает в Docker и использует микросервисную архитектуру для управления параллельными задачами, синхронизацией состояний и безопасным привилегированным доступом к аппаратному обеспечению (дискам):

```
                  ┌──────────────────────────────────────────────┐
                  │              React SPA Frontend              │
                  │             (Port 7777 - Nginx)              │
                  └──────────────────────┬───────────────────────┘
                                         │ REST API
                                         ▼
                  ┌──────────────────────────────────────────────┐
                  │               FastAPI Backend                │
                  │             (Port 8000 - Uvicorn)            │
                  └──────────────┬───────────────┬───────────────┘
                                 │               │
                     Writes Logs │               │ Dispatches Tasks
                     & Metadata  │               │
                                 ▼               ▼
   ┌──────────────┐       ┌──────────┐      ┌──────────┐       ┌─────────────┐
   │  PostgreSQL  │ ◄──── │ Database │      │  Redis   │ ◄───► │   Celery    │
   │ (Port 5432)  │       │  Session │      │  Broker  │       │ Task Worker │
   └──────────────┘       └──────────┘      └──────────┘       └──────┬──────┘
                                                                      │ Runs
                                                                      │ Privileged Actions
                                                                      ▼
                                                               ┌─────────────┐
                                                               │  Edge Fleet │
                                                               │   Targets   │
                                                               └─────────────┘
```

### Компоненты:
1. **React SPA Frontend (Порт 7777)**: Адаптивная панель управления с темной темой, разделенная на вкладки (Fleet, Flasher, History, Live-USB Client, Settings). Поддерживает мультиязычный интерфейс (английский, русский, украинский) с премиальным анимированным селектором языков. Отображает статистику (коэффициент дедупликации, занятое место) и включает оверлей терминала для стриминга логов выполнения задач в реальном времени.
2. **FastAPI Backend (Порт 8000)**: Предоставляет RESTful API, включает продвинутый парсер IP-адресов (с поддержкой CIDR, списков и диапазонов), проверяет конфигурации типов дисков и отслеживает активные задачи.
3. **Celery Worker (Привилегированный доступ к хосту)**: Подписан на очередь задач. Выполняет Ansible-плейбуки и низкоуровневые команды прошивки разделов (требует проброса `/dev` хост-системы оркестратора для прошивки дисков).
4. **Borg SSH Server (Порт 12345)**: Изолированная среда центрального репозитория, куда публичные ключи узлов автоматически добавляются в `/home/borg/.ssh/authorized_keys` со строгими ограничениями выполняемых команд (`command="borg serve --restrict-to-path ..."`).
5. **Redis**: In-memory брокер очередей и бэкенд для хранения результатов Celery.
6. **PostgreSQL**: Хранит состояния оркестратора, глобальные настройки, метаданные инвентаризации узлов, историю бэкапов и логи выполнения задач.

## 🛠️ Ключевые модули оркестрации

### 1. Добавление парка (Fleet Provisioning) и массовый парсинг IP
- Поддерживает регистрацию хостов через **списки** (через запятую), **диапазоны** (например, `192.168.1.50-60`) и **CIDR-подсети** (например, `192.168.1.0/30`).
- Запускает параллельные задачи Celery (по умолчанию до 24 потоков) для одновременной быстрой подготовки множества комплексов.
- Поля формы предзаполнены дефолтными учетными данными (`user`, `admin`, `SSH port 2222`) для ускорения рутинной работы.

### 2. Плейбук автоподготовки (Auto-Prepare)
- Выполняет идемпотентный Ansible-плейбук для проверки готовности узла.
- Присваивает постоянные метки файловым системам (`edgeroot` для корня, `edgeboot` для `/boot` и т.д.).
- Считывает и сохраняет уникальный аппаратный UUID FAT32-раздела EFI.
- Перезаписывает `/etc/fstab` на целевом устройстве для монтирования строго по меткам. Это защищает систему от смещения имен дисков в Linux (например, когда при смене железа `/dev/sda` превращается в `/dev/nvme0n1`).

### 3. Планирование бэкапов, дедупликация и лимиты ресурсов
- Бэкапы инициируются удаленно командой через `ssh` и стримят данные на центральный Borg SSH Server.
- **SSH-активность (Keepalive)**: Для исключения разрывов SSH-сессий на нестабильных или медленных каналах связи все исходящие команды SSH и переменная `BORG_RSH` снабжены keepalive-параметрами (`ServerAliveInterval` и `ServerAliveCountMax`), настраиваемыми через переменные окружения в `.env`. Это страхует фоновые задачи от бесконечного зависания.
- **Гибкие лимиты ресурсов**: Для защиты производительности узлов и снижения утилизации каналов связи резервное копирование поддерживает следующие лимиты:
  - **Лимит скорости (Upload Rate Limit)**: Максимальная скорость передачи данных (в KiB/s), настраиваемая для группы бэкапа.
  - **Ограничение CPU (CPU Quota)**: Ограничение использования процессора (в % от 1 ядра, 0-400%) на стороне клиентского узла, применяемое глобально или для группы через `systemd-run --scope -p CPUQuota=...`.
  - **Кастомное сжатие**: Настройка алгоритма сжатия (lz4, zstd:3 по умолчанию, zstd:5 и др.) глобально во вкладке настроек или индивидуально для группы бэкапов.
  - **Оптимизированный интервал чекпоинтов**: Частота создания точек восстановления Borg рассчитывается автоматически на основе лимита скорости (каждые ~50 МБ при скорости <= 500 KiB/s, ~200 МБ при <= 5000 KiB/s или стандартные 1800 секунд при высоких скоростях) для оптимального сохранения данных при обрывах связи, либо переопределяется вручную.
- **Сжатие и сквозная дедупликация**: Поскольку все комплексы сохраняются в единый центральный репозиторий Borg (`/data/borg/fleet`), сжатие и дедупликация на уровне чанков работают глобально по всему парку устройств. Одинаковые файлы, системные бинарники ОС и данные приложений, присутствующие на разных узлах Debian, сохраняются на сервере **только один раз**.
  - *Пример экономии места (при размере базовой ОС ~6 ГБ на комплекс)*:
    - **1-й комплекс (Сжатие и внутренняя дедупликация)**: с ходу экономит **55% - 65%** дискового пространства благодаря архивированию Borg (по умолчанию `lz4`), сжимая 6 ГБ системы до ~2.2 - 2.7 ГБ фактического объема на диске.
    - **Каждая последующая копия схожей системы (Сквозная дедупликация)**: экономит до **97%** для клонов и идентичных устройств (добавляя всего **~100 - 200 МБ** для того же устройства под другим хостнеймом) и около **20% - 30%** для комплексов с небольшими различиями в пакетах и конфигурации.
    - **Последующие инкрементные бэкапы**: работающих систем стремятся к небольшому объему около **~100-200 МБ** уникальных инкрементных данных на один запуск бэкапа (сохраняются только изменившиеся логи, кэши, базы данных).
    - Это позволяет радикально снизить общий объем занимаемого дискового пространства для всего парка однотипных систем, при этом инкременты остаются крайне легкими.
- **Настраиваемые исключения (Exclusions)**: Через вкладку настроек (**Orchestrator Settings**) в веб-интерфейсе можно гибко задать список директорий (через запятую), которые нужно исключить из резервного копирования (например, виртуальные файловые системы или директории с тяжелыми логами/данными).
  - **Список исключений по умолчанию**: `/dev/*,/proc/*,/sys/*,/run/*,/mnt/*,/media/*,/lost+found,/var/log/edge/*,/var/opt/edge/*`
- Для предотвращения блокировок репозитория, ротация архивов (pruning) отделена от самого процесса резервного копирования. Глобальный планировщик Celery Beat ежедневно в 3:00 ночи запускает `borg prune` с использованием политики удержания (настраивается глобально во вкладке Settings или переопределяется индивидуально для группы бэкапов). Поддерживаются три режима ротации: по интервалам (дневные/недельные/месячные лимиты), по количеству (хранить последние N копий) или по временному окну (хранить за последние N дней/недель/месяцев/лет).

### 4. Восстановление (Bare-Metal Flashing Restore)
- **Защита от перезаписи хоста**: Оркестратор сканирует блочные устройства хоста и жестко фильтрует собственный системный диск, исключая случайную перезапись "самого себя".
- **Drive Type Mismatch Warning & Cross-Drive Migration**: Благодаря тому, что система строго использует метки файловых систем (`LABEL=edgeroot`) в `/etc/fstab` вместо аппаратных путей `/dev/sdX` или UUID, **вы можете бесшовно восстановить бэкап, снятый с NVMe-накопителя, на SATA-диск** (и наоборот). При обнаружении смены типа диска веб-интерфейс выдаст предупреждение "by-design", чтобы вы были в курсе, но позволит подтвердить и продолжить кросс-миграцию.
- **Сохранение EFI UUID**: Переразмечает целевой USB-накопитель в GPT, форматирует загрузочный раздел и явно переопределяет его UUID исторически сохраненным значением (`mkfs.vfat -i <EFI_UUID_HEX>`).
- **Защита от смещения сетевых интерфейсов**: Удаляет старые привязки сетевых карт (`persistent-net`) и внедряет универсальные настройки (для `eth*` и `en*`), гарантируя появление сети после установки диска в новое оборудование.
- **Конфигурация загрузчика через Chroot**: Монтирует систему, пробрасывает виртуальные пути (`/dev`, `/proc`, `/sys`), переустанавливает GRUB на целевом диске, обновляет `initramfs` и создает резервный путь EFI загрузчика (`EFI/BOOT/BOOTX64.EFI`).
- **Аудит**: После восстановления выполняет проверку конфигурации меток в `/etc/fstab` перед безопасным отключением.

### 5. Генерация Live-USB клиента
- На лету компилирует загрузочную среду Debian Live.
- Встраивает IP-адрес оркестратора и токены авторизации напрямую в генерируемый ISO-образ.
- При загрузке на клиентском узле автоматически запускает защищенный киоск, который подключается к центральному оркестратору и позволяет восстанавливать систему по сети прямо на внутренний диск узла без его извлечения.

## 🔌 Стабильность оборудования и USB-накопителей

При выполнении операций "bare-metal restore" на целевые USB-флешки или внешние жесткие диски, подключенные через мосты USB-to-SATA/NVMe (например, контроллеры JMicron, такие как `152d:0581`), стандартный Linux-драйвер **UAS (USB Attached SCSI)** может зависать при высоких нагрузках конкурентной записи (характерно для `borg extract`). Из-за этого процесс записи намертво зависает в статусе `D` (I/O wait).

Для гарантии стабильности хоста при прошивке:

1. **Отключите UAS для вашего контроллера** на хост-машине (NUC), принудительно переключив его на надежный legacy-драйвер `usb-storage`:
   ```bash
   echo -e "options usb-storage quirks=152d:0581:u\noptions uas quirks=152d:0581:u" | sudo tee /etc/modprobe.d/usb-quirks.conf
   ```
   *(Замените `152d:0581` на Vendor:Product ID вашего переходника, узнать который можно командой `lsusb`).*

2. **Переподключите USB-устройство** для применения правила, либо сбросьте шину USB программно:
   ```bash
   sudo usbreset 152d:0581
   ```

## 🔐 Безопасность и авторизация пользователей

Для защиты панели управления и API оркестратора Edge B.R.O. реализует ролевую модель доступа (RBAC):

1. **Авторизация в панели**:
   - Доступ к административной панели требует входа по имени пользователя и паролю.
   - Сессии пользователей управляются с использованием безопасных JWT-кук с флагом HTTP-only (`admin_session`), что защищает от межсайтового скриптинга (XSS).

2. **Роли пользователей**:
   - **Superadmin (Суперадминистратор)**: Главная учетная запись администратора. Может управлять другими учетными записями администраторов во вкладке **Настройки → Администраторы** (создавать, редактировать данные, устанавливать/сбрасывать пароли, удалять и добавлять комментарии).
   - **Administrator (Администратор)**: Обычная учетная запись администратора. Может просматривать узлы, задачи, настройки, запускать бэкапы/восстановление, но не может управлять пользователями.

3. **Технические киоски (Technician Kiosks)**:
   - Авторизованные технические киоски, загруженные с Live-USB, обходят стандартную страницу входа, отправляя ранее привязанный токен в заголовке `Authorization: Bearer <auth_token>`.
   - Для offline-восстановления с ISO-образов, где сетевые настройки не заданы, поддерживается авторизация через параметры запроса (`?token=<auth_token>`).

4. **Первоначальное заполнение БД и восстановление**:
   - При первом запуске оркестратор автоматически создает учетную запись суперадминистратора по умолчанию, используя значения из переменных окружения (`.env`):
     - Имя пользователя: `SUPERADMIN_USERNAME` (по умолчанию `admin`, если не задано)
     - Пароль: `ADMIN_PASSWORD` (по умолчанию `admin_pass`, если не задано)
   - Эти учетные данные создаются в базе данных только один раз. Последующие изменения в файле `.env` не будут перезаписывать пароль, измененный через веб-интерфейс.
   - Если вам нужно сбросить пароль суперадминистратора, измените значения в `.env` и перезапустите контейнер, либо очистите таблицу `users` в базе данных для повторной инициализации.
